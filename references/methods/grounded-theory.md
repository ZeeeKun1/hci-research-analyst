# Grounded Theory

## Overview

Grounded Theory (GT) is a methodology for developing theory that is "grounded in" data, rather than testing pre-existing theory. Two main schools: Glaserian (emergent, minimalist) and Straussian (more structured with explicit coding paradigm). Charmaz (2006) offers a constructivist revision widely adopted in HCI.

## Key Distinguishing Features from Thematic Analysis

- TA → identifies themes; GT → builds theory
- GT uses theoretical sampling (collect data based on emerging categories)
- GT aims for theoretical saturation (not data saturation)
- GT produces a core category + theoretical framework, not just a list of themes

## Three Versions

| School | Approach | Best For |
|--------|----------|----------|
| Glaser (1967) | Let theory emerge; avoid forcing | When genuinely exploring new territory |
| Strauss & Corbin (1990/1998) | Structured coding paradigm; memoing | When some structure is helpful |
| Charmaz (2006) | Constructivist; acknowledges researcher role | HCI research with interpretive lens |

*This guide follows Charmaz's constructivist approach as default for HCI.*

## Core Procedures

### 1. Initial Coding (Line-by-Line)
- Code every data segment that is relevant.
- Use gerunds ("struggling with interface," "anticipating system response") to capture actions/processes.
- Stay close to the data; avoid abstract codes at this stage.
- Ask: "What is happening here?" "What is the participant doing/saying?"

### 2. Focused Coding
- Review initial codes; select the most significant and frequent ones.
- Synthesize initial codes into larger, more conceptual categories.
- Use focused codes to sort and organize data.

### 3. Axial Coding
- Relate categories to subcategories.
- Use the coding paradigm (Strauss): Conditions → Strategies → Consequences.
- For each category, identify:
  - Causal conditions: What leads to this?
  - Context: What conditions shape it?
  - Intervening conditions: What facilitates/constrains it?
  - Strategies: How do actors respond?
  - Consequences: What are the outcomes?

### 4. Theoretical Coding
- Integrate categories around a core category.
- The core category is the central phenomenon that explains most variation.
- Subcategories relate to the core category through:
  - Temporal sequence
  - Causal relationships
  - Means-end chains
  - Type (substantive categories within core)

### 5. Memo Writing
- Write memos throughout the process. Memos are analytic notes that:
  - Capture emerging ideas
  - Develop conceptual density
  - Track decisions and reasoning
  - Link data to categories
- Memos become the foundation of the written analysis.

### 6. Theoretical Sampling & Saturation
- Seek data that elaborates and refines emerging categories.
- Continue until categories are saturated (no new properties emerge).
- NOT about collecting all possible data — about conceptual density.

### 7. Sorting & Writing
- Sort memos and data into the theoretical framework.
- Write the theory with sufficient grounding in data extracts.

## HCI-Specific Guidance

- **Technology as actor**: In HCI, technology often plays an active role. Code for how the technology shapes actions, not just user behavior.
- **Process orientation**: GT is inherently processual. Capture how experiences unfold over time.
- **Comparative analysis**: Constantly compare between participants, contexts, and system states.
- **Actionable theory**: HCI values theory that can inform design. Grounded theory can produce design implications directly from data.

## When to Choose GT Over TA

Choose GT when:
- You want to BUILD theory, not just identify patterns.
- You have theoretical sampling flexibility (can collect more data).
- The phenomenon is poorly understood and needs conceptual development.
- Process and sequence are central to understanding.

Choose TA when:
- You want to describe patterns within a known area.
- You have fixed data and cannot collect more.
- Theory testing/application is more relevant than theory building.
