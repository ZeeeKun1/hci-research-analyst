# Thematic Analysis (Braun & Clarke)

## Overview

Thematic Analysis (TA) is a flexible, accessible method for identifying, analyzing, and reporting patterns (themes) within data. The Braun & Clarke (2006) 6-phase approach is the most widely used variant in HCI research. It is NOT limited to qualitative data and can be applied at different theoretical levels.

## Theoretical Levels

Clarify which level you're operating at:
1. **Reflexive TA**: Researcher interpretive lens is central; themes are developed organically from data. Most common in HCI.
2. **Coding Reliability TA**: Uses codebook + multiple coders; aims for inter-coder agreement. Common in survey-based qualitative analysis.
3. **Codebook TA**: Pre-developed codebook applied to data; deductive. Common when testing a priori frameworks.

*This guide covers Reflexive TA as default; note adjustments for other variants where relevant.*

## Six Phases

### Phase 1: Familiarization with the Data
- Read and re-read all data. Read holistically, then analytically.
- Note initial observations, striking features, patterns that jump out.
- If data is qualitative: pay attention to language, emotion, repetition, omissions.
- If data is quantitative-survey: note response patterns, surprising answers, open-ended comments.
- **Output**: Initial notes, not yet systematic.

### Phase 2: Generating Initial Codes
- Code systematically across the entire dataset.
- A code captures a single interesting feature of the data relevant to the research question.
- Codes can be semantic (explicit surface meaning) or latent (underlying ideas, assumptions).
- Use both data-driven (inductive) and theory-driven (deductive) coding.
- **Principle**: Code at the level of meaning, not just topic. "User mentioned difficulty" is a topic code; "User framed difficulty as personal inadequacy rather than system flaw" is a meaning-level code.
- **Output**: A list of codes with supporting data segments.

### Phase 3: Generating Initial Themes
- Collate codes into potential themes.
- Each theme should capture a coherent pattern of meaning across the dataset.
- Consider: What does this collection of codes tell us about the data?
- Create a thematic map (visual) showing relationships between themes.
- At this stage, themes are tentative. Expect revision.
- **Output**: Thematic map + theme descriptions with supporting codes.

### Phase 4: Reviewing Themes
**Two levels of review:**

*Level 1: Check themes against coded data*
- Does the data within each theme cohere?
- Are there codes that don't fit? Move or remove them.
- Are there distinct themes that should be merged?

*Level 2: Check themes against the entire dataset*
- Does each theme accurately reflect the dataset as a whole?
- Generate a candidate thematic map and check against complete data.
- Recode if necessary (it's normal to cycle back to Phase 2).

- **Output**: Refined thematic map with verified theme-data alignment.

### Phase 5: Defining and Naming Themes
- For each theme, write a detailed analysis:
  - What is the story of this theme?
  - What does it tell us about the research question?
  - What is the scope and boundaries of this theme?
  - What are the nuances and complexities?
- Name themes clearly — the name should capture the essence.
- Generate a thematic table: Theme → Definition → Supporting evidence.

### Phase 6: Producing the Report
- Write up the analysis with selections from the data to support each theme.
- Balance analytic narrative with vivid, illustrative data extracts.
- Connect themes back to the research question and literature.
- Acknowledge limitations, alternative interpretations, and negative cases.

## HCI-Specific Guidance

- **User quotes**: Present with participant ID, context note if needed. Avoid long block quotes; select the most illustrative segments.
- **System interaction data**: Code behavioral patterns alongside verbal reports for triangulation.
- **Technology-specific codes**: Include codes that capture the technology's role (e.g., "system-as-agent," "technology-as-barrier," "affordance exploration").
- **Negative space**: What participants did NOT mention or do can be as informative as what they did. Document notable absences.

## Common Pitfalls

| Pitfall | How to Avoid |
|---------|-------------|
| Too many themes (>15) | Themes should tell a coherent story. Consolidate. |
| Too few themes (<3) | Look for sub-themes, contradictions, boundary cases. |
| Code-theme mismatch | Themes must be supported by multiple codes; codes by multiple data extracts. |
| Cherry-picking quotes | Show the range of evidence, including contradictory data. |
| Descriptive, not analytic | Go beyond "users said X" → explain what X means in context. |
